Building Tensorbit Core: A C++/CUDA Transformer Pruning Library

C++ CUDA Model Optimization LLM Sparsity

May 2026 · 15 min read

Large language models keep getting bigger. Every new release pushes the parameter count higher, and with it the compute and memory requirements for inference. The standard response is to buy more GPUs. The better response is to prune.

Tensorbit Core is a pruning library I built as part of Tensorbit Lab that applies structured and semi-structured sparsity to transformer architectures at the CUDA level. This writeup covers the motivation, the algorithms (EHAP and CORING), the engineering decisions, and what I learned building a systems-level ML library from scratch in C++.

Motivation: Why Prune?

Most of the weights in a trained neural network are redundant. Research has shown that you can remove 50-90% of parameters with minimal accuracy loss. Pruning is the practice of zeroing out these redundant weights to produce a sparse network that is smaller, faster, and more power-efficient.

Existing pruning libraries like SparseGPT and Wanda work well but are designed for post-training pruning on already-trained models. Tensorbit Core takes a different approach: it integrates pruning into the model life cycle as a first-class operation, supporting both one-shot and iterative pruning with recovery fine-tuning.

EHAP: Efficient Hessian Aware Pruning

The core idea behind EHAP is to use second-order information to decide which weights to prune. Instead of looking at weight magnitudes alone (which assumes all weights are equally important), EHAP approximates the Hessian to measure how much the loss changes when a weight is removed.

The exact Hessian is prohibitively expensive to compute for large models. EHAP uses an efficient approximation: it computes the empirical Fisher information matrix using a calibration dataset, then uses the diagonal entries as a proxy for weight importance. Weights with low importance scores are pruned first.

// Simplified EHAP scoring
for each layer in model:
    features = layer.forward(calibration_data)
    hessian_diag = sum(features.grad ** 2, dim=0)
    importance = (weights ** 2) / (hessian_diag + EPS)
    mask = importance > threshold(importance, sparsity_target)
    weights *= mask

CORING: Coarse-to-Fine Iterative Pruning

CORING is the scheduling algorithm that controls how pruning progresses across layers. The insight is that different layers have different sensitivity to pruning. Early layers (closer to input) tend to be more sensitive; later layers can tolerate higher sparsity.

CORING allocates sparsity targets to each layer adaptively: it starts with conservative targets, evaluates the loss impact, then redistributes the budget from less-sensitive to more-sensitive layers in subsequent iterations. Each iteration prunes a small percentage of weights, then runs a short recovery fine-tuning step.

This coarse-to-fine approach avoids the sharp accuracy drops that come with one-shot high-sparsity pruning. In practice, we found that three iterations with recovery between each was sufficient to reach 60% unstructured sparsity with less than 1% accuracy degradation on a BERT-base model.

CUDA Implementation

The pruning operations themselves run on GPU via custom CUDA kernels. The key kernel is a sparse-dense matrix multiplication that applies the binary mask during the forward pass. Instead of storing the full weight matrix and masking at runtime (which saves no compute), we use a compressed sparse row format that skips zeroed weights entirely.

// CUDA kernel: sparse-dense matmul with CSR format
__global__ void csrmm_kernel(
    const float* values, const int* col_indices, const int* row_ptr,
    const float* input, float* output,
    int M, int N, int K
) {
    int row = blockIdx.x * blockDim.x + threadIdx.x;
    if (row >= M) return;
    float sum = 0.0f;
    for (int j = row_ptr[row]; j < row_ptr[row + 1]; j++) {
        int col = col_indices[j];
        sum += values[j] * input[col];
    }
    output[row] = sum;
}

The library also includes a custom autograd function for PyTorch integration, allowing pruning-aware training with standard PyTorch optimizers and schedulers.

Results and Benchmarks

We benchmarked Tensorbit Core against SparseGPT and Wanda on BERT-base and a small GPT-2 variant. At 50% unstructured sparsity, EHAP + CORING matched SparseGPT's perplexity while being 3x faster to compute (since it doesn't require the full Hessian inversion that SparseGPT does). At 70% sparsity, the accuracy gap widened slightly in SparseGPT's favor, but CORING with one recovery fine-tuning epoch closed the gap entirely.

The practical takeaway: if you can afford one recovery fine-tuning pass, EHAP + CORING matches or exceeds one-shot methods while being significantly cheaper to compute.

Lessons Learned

  • CUDA debugging is hard. Learning to use cuda-gdb and compute-sanitizer was essential. Most bugs were memory access violations that only showed up at specific matrix sizes.
  • Hessian approximation quality matters more than precision. A cheap diagonal approximation worked as well as more expensive block-diagonal methods for our use case.
  • Integration with PyTorch's autograd is worth the effort. Being able to fine-tune with optimizer.step() after pruning made iterative pruning much easier to implement.
  • Start with a single layer, then generalize. Debugging pruning logic on a 12-layer model is painful. Getting it right on one layer first saved hours.
